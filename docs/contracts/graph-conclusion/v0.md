# Graph Conclusion Contract V0

## Purpose

This contract defines how one completed Cadenza graph run exposes a deterministic result without changing the graph runner's primary responsibility for coordination.

The conclusion is an internal run-scoped composition boundary. It is not an authored task, a persisted graph node, or a chamber transport concern.

## Identities

- **run baseline**: the immutable user context supplied when the graph run begins.
- **completion participant**: an instantiated runtime branch that must finish before the run concludes.
- **result contributor**: a completed terminal runtime branch whose context participates in automatic composition.
- **terminal provenance**: the task key, task execution identity, and runtime graph path that produced a terminal context.
- **graph conclusion**: the single success or failure produced after every completion participant finishes.

All deltas are calculated against the one run baseline. A split inside the run never creates a new composition baseline.

## Runtime Conclusion Boundary

The core creates one internal conclusion tracker for each graph-run identity. Every actually instantiated terminal branch registers with it. The tracker waits for every instantiated completion participant, including branches excluded from result contribution.

The conclusion tracker must not be represented by mutating authored task relationships. Static graph analysis may cache topology, but runtime participation remains authoritative because task outcomes can stop branches before their statically reachable successors are instantiated.

## Relationship Contribution Policy

Every task relationship has a result-contribution policy:

```json
{
  "contributes_to_result": true,
  "target_task_key": "task.example.next"
}
```

The default is `true`. Setting it to `false` excludes changes reached exclusively through that relationship lineage from automatic result composition.

The policy does not detach work and does not remove the branch from completion. Work that should neither contribute nor delay the current graph must cross a signal boundary.

If excluded and contributing lineages converge in an authored join task, that task's output is contributing. The authored join owns the meaning of its result.

## Provenance Delta

Only user-context fields participate in composition. Runtime metadata keys beginning with `__` are carried by the core and are not compared as business result paths.

For each contributing terminal, the core derives a recursive delta from the run baseline:

- plain JSON objects are compared recursively by key.
- arrays are atomic values.
- scalars and `null` are atomic values.
- an absent terminal key that existed in the baseline is a deletion.
- an added key, changed scalar, changed array, or object-to-non-object transition is a write.
- unchanged paths produce no delta.

Paths use RFC 6901 JSON Pointer encoding. Object keys and terminal provenance are ordered canonically before comparison or reporting.

## Deterministic Composition

- disjoint path changes merge.
- canonically equal writes to the same path collapse.
- identical deletions of the same path collapse.
- different writes to the same path conflict.
- deletion and write of the same path conflict.
- a write or deletion of a parent path conflicts with a different mutation below that path.
- completion order never resolves a conflict.

The merged delta is applied to a clone of the run baseline. The baseline itself is never mutated.

## Authored Join Tasks

Automatic composition handles only unambiguous structural changes. An architect who needs domain-specific precedence, reduction, aggregation, or reconciliation defines an ordinary unique join task and connects the relevant branches to it.

When that authored join is the contributing terminal, the internal conclusion forwards its resulting context. The core does not reinterpret the authored merge.

## Outcomes

```text
completed
  composed context

composition_failed
  deterministic conflict paths and terminal provenance

execution_failed
  deterministic failed terminal provenance
```

Raw conflicting values are not required in ordinary diagnostics. Implementations may expose them only through an explicitly authorized diagnostic surface.

A composition failure is an execution failure at a chamber delegation boundary. It must never be reported as `fulfilled`, silently merged, or safely retryable after execution has started.

## Required Scenarios

- a linear graph forwards its sole terminal context.
- disjoint terminal deltas merge against the run baseline.
- identical terminal writes collapse.
- conflicting terminal writes produce `composition_failed` independent of completion order.
- an excluded branch is awaited but does not contribute.
- a signal-detached branch is neither awaited nor included.
- an authored unique join supplies the terminal result without a second implicit merge.
- a failed terminal produces `execution_failed` rather than a partial fulfilled result.

## Non-Goals

- defining durable result persistence.
- adding result ownership to chambers or cells.
- serializing callable implementations.
- defining domain-specific merge precedence.
- treating intermediate split contexts as composition baselines.
