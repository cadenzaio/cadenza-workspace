# Actor Core Contract V1

## Purpose

This contract defines the language-neutral semantic actor baseline for Sprint
8A. It does not define distributed residency, persistence, ownership, routing,
or recovery.

An actor is a keyed state authority. It is not executable and it does not own
tasks. Tasks remain ordinary executable primitives and declare actor access on
the task side through `cadenza.actor.binding.v1`.

## Actor Definition

Canonical JSON uses:

```json
{
  "schema_version": 1,
  "actor_key": "actor.counter",
  "name": "Counter",
  "description": "Counts by account",
  "keying": {
    "default_key": "counter.default",
    "resolver": { "source": "field", "field": "account_id" }
  },
  "state": {
    "durable": {
      "init_state": { "count": 0 },
      "schema": { "type": "object" }
    },
    "runtime": {
      "schema": { "type": "object" }
    }
  },
  "extensions": []
}
```

Rules:

- `actor_key` is the actor definition identity; a runtime state key is a
  separate identity.
- `name` is the unique semantic name within its authority scope.
- persisted `init_state` is data, not callable source.
- key resolution is declarative in definitions. Language APIs may accept a
  local callable resolver, but it is not serialized as semantic authority.
- extension ids and orders are unique and extension order is deterministic.
- unsupported top-level fields are rejected; future behavior enters through a
  versioned extension rather than an ad hoc actor field.
- task bindings, load policy, write contract, retry, idempotency, consistency,
  meta classification, runtime read guards, persistence, and residency are not
  actor semantic fields.
- materialization is lazy. Reading state through inspection may materialize a
  local key, but definitions do not request eager loading.

## Task Binding

The canonical task-side extension is:

```json
{
  "id": "cadenza.actor.binding.v1",
  "order": 20,
  "config": {
    "actor_key": "actor.counter",
    "access": "write",
    "role": "user",
    "interaction": "local_residency"
  }
}
```

Allowed values:

- `access`: `read` or `write`.
- `role`: `user` or `manager`.
- `interaction`: `local_residency` or `signal_mediated`.

Each task has at most one actor binding. The binding references actor authority,
not a state key, version, owner, route, endpoint, or persistence provider.

Manager bindings are meta-layer bindings. `meta` is not an actor access mode.

## Local Execution

- state keys resolve in this order: explicit invocation state key, local
  callable resolver, declarative resolver, default key.
- invocation-only actor options do not enter business input.
- reads and writes for one state key execute in one strict sequence.
- different state keys may execute concurrently.
- read bindings cannot use state mutators.
- direct mutation of supplied state snapshots does not mutate actor authority.
- a failed handler does not commit candidate local state.
- each successful execution that changes durable or runtime state advances the
  corresponding version once.
- runtime state is local and is not part of persisted actor semantic state.

## Boundary

The core receives already materialized handlers. Callable source conversion is
a Cell/Chamber adapter responsibility. Persistence and distributed actor
lifecycle are Sprint 8B-8D responsibilities and must enter through hidden
runtime coordination contracts rather than public actor semantics.

## Conformance

Every official core must consume the canonical fixture in
`fixtures/v1-actor-core.json` and prove:

- exact semantic definition round-trip.
- exact task-binding meaning.
- rejection or absence of transitional actor fields.
- strict same-key sequencing and independent-key concurrency.
- failed-write atomicity and read-write enforcement.

Language-specific APIs may be idiomatic but cannot weaken these meanings.
