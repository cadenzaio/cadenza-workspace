# Cell Peer Transport Contract V0

## Purpose

This contract carries ordinary Cadenza delegation and signal behavior directly
between two enrolled cell generations. Distribution remains invisible to
chambers and authored business definitions: chambers select opaque route-member
keys and the source cell resolves the selected member to local or peer ingress.

V0 defines direct one-hop transport only. It does not define discovery, relay,
multi-hop routing, brokers, automatic failover, exactly-once execution, or
concurrent chamber execution.

## Authority

- `cadenza` owns cell enrollment, assignment, residency, and derived
  route-member meaning.
- `cadenza-chamber` owns the opaque route callback and normalized primitive
  outcomes.
- `cadenza-cell` owns local/remote interpretation, peer key custody, TLS,
  signed envelopes, replay state, and target ingress enforcement.

## Transport Identity

Each cell has a distinct enrolled transport key. `transport_key_digest` is the
lowercase `sha256:` digest of the certificate's DER SubjectPublicKeyInfo, not a
digest of the complete certificate. The cell verifies that its Ed25519 PKCS#8
private key matches the public key in its X.509 certificate.

TLS requirements are:

- TLS 1.3 only.
- mutual certificate authentication.
- exact configured peer certificate trust roots.
- post-handshake SPKI comparison with current active cell enrollment.
- current transport key reference and protocol equality.
- fixed authority endpoint for the target cell.
- no plaintext, opportunistic TLS, ambient platform roots, or bearer token
  fallback.

Peer configuration is enrollment-scoped: it carries the stable Cell key,
transport public-key reference, certificate, and derived SPKI digest. It does
not predict or pin a process-generation key. After TLS authentication, the
receiver reads distribution authority and requires the generation claimed by
the signed `open` control to be the exact current unexpired ready generation
for that enrolled Cell. The verified session then binds that generation across
execution, proceed, result, and evidence. Restart therefore creates fresh
generation identity without requiring a retained peer process to restart or
receive mutable peer configuration.

## Signed Control

Before an execution envelope, peers exchange signed `open` and `opened`
controls. Each control binds:

- contract and protocol version.
- fresh session and nonce identity.
- source environment, cell, and generation.
- exact target cell and generation.
- transport public-key reference.
- issue and expiry time no more than 30 seconds apart.
- response nonce where applicable.

Signatures are Ed25519 over bounded canonical JSON with the
`cadenza.cell-peer.v0` domain. TLS peer authentication does not replace these
durable signed identities.

## Execution Envelope

A signed envelope binds the session and a fresh envelope key, source and target
cell generations, deadline, source transport-key identity, and exactly one
typed operation:

- delegation: complete route decision and chamber-originated route delegation
  request.
- signal: complete route decision and chamber-originated route signal request.

The route decision carries unit and definition revision, replica and assignment
epoch, route-member identity, target cell generation, slice, lane, chamber,
image digest and epoch, and task or signal responsibility. It contains no
endpoint or credential.

Route epoch remains source cell projection state and is signed in the request.
At target ingress, the cell rereads current route members for that responsibility
and epoch and requires every durable placement field to equal the signed route
decision. The target then independently requires the exact local generation,
bound route member, chamber image, lane, readiness, and projection eligibility.

## Start Boundary

Execution uses a reserve/accept/proceed sequence:

1. the target authenticates TLS, controls, envelope signature, lifetime,
   bounds, current enrollment, the claimed source generation as exact current
   ready authority, current route authority, and replay identity.
2. the target reserves the envelope and sends signed `accepted`.
3. the source verifies `accepted` and sends signed `proceed`.
4. only after valid `proceed` does the target mark the envelope started and
   invoke the chamber.
5. the target returns a signed normalized result.

Failure before `proceed` is `execution_started: false`. Once `proceed` is sent,
connection loss or an absent result is conservatively
`execution_started: true`; it is never presented as safely retryable.

## Bounds And Replay

- peer frames: at most 2 MiB, length checked before allocation.
- business context: at most 1 MiB.
- delegation result bound: positive and at most 1 MiB in V0.
- control/envelope lifetime: at most 30 seconds and bounded by the authored
  deadline.
- replay ledger: at most 16,384 generation-local entries.
- duplicate envelope identity and invalid replay transitions fail closed.

The target does not claim durable exactly-once execution across cell loss.
Duplicate envelopes are rejected; V0 does not return cached business results or
turn the transport replay ledger into a persistence layer. Retry policy remains
outside this direct transport contract.

## Transport Evidence

Every operation emits canonical, digest-bound evidence through a mandatory
append-only host sink. The target records `target_accepted`, `target_started`,
and `target_completed`; the source records `source_outcome`. Records bind both
cell generations, operation, correlation and route-member identities, outcome,
started state, and normalized failure code where present.

Transport evidence never contains business context, initiating subject,
runtime principal, credentials, certificates, keys, or endpoints. Failure to
append required evidence fails the operation closed. This transport evidence is
distinct from the deferred realtime per-task business-execution evidence
contract.

## Cell Host Process

The Linux `cadenza-cell` host must run unprivileged and accepts only these fixed
inherited descriptors:

- `3`: bounded canonical host config.
- `4`: DER transport certificate.
- `5`: Ed25519 PKCS#8 transport private key.
- `6`: PostgreSQL distribution-reader credential restricted in V0 to an
  explicit Unix socket or loopback host because the existing reader uses
  `NoTls`.
- `7`: pre-bound peer listener.
- `8`: typed lifecycle control channel.
- `9`: bounded canonical append-only peer transport evidence channel.

The process accepts no path, shell, arbitrary command, mount, provider, peer
destination, or environment credential. Its listener must equal current cell
enrollment. Sprint 6C adds authority-driven chamber activation to this process;
Sprint 6B proves its identity, custody, lifecycle, and transport boundary.
Remote PostgreSQL access requires a separately approved authenticated TLS
contract and cannot be enabled through this descriptor.
