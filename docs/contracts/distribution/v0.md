# Multi-Cell Distribution Contract V0

## Purpose

This contract defines the authority by which one environment enrolls multiple
cells, declares placement intent, commits deterministic reconciliation plans,
applies exact assignments, converges local Chambers, observes ready replicas,
and derives direct route members. It also defines the identities carried by
authenticated Cell transport and the first contained stem-cell policy loop.

It also defines pre-enrolled local Cell supply and fenced singleton stem
recovery. It does not define dynamic enrollment, machine or cloud provisioning,
multi-hop transport, actor residency, or generalized durable orchestration.

## Authority

- `cadenza` owns source, enrollment, placement, and authority-operation meaning.
- `cadenza-cell` owns cell-generation state, local enforcement, residency
  publication, route interpretation, and later peer transport.
- `cadenza-chamber` owns runtime projections and primitive ingress but does not
  own cell topology or placement authority.

## Canonical Data

Cross-repo data uses bounded canonical JSON:

- printable ASCII object keys sorted by ascending byte value.
- UTF-8 string values containing Unicode scalar values.
- arrays preserve declared order; set-like arrays are sorted and duplicate-free.
- integers are positive and no greater than `9007199254740991`.
- floats and exponential numbers are forbidden.
- digests use `sha256:` plus 64 lowercase hexadecimal characters.
- signatures use `base64:` followed by canonical base64 Ed25519 bytes, matching
  the existing bootstrap and activation contracts.
- timestamps crossing the runtime boundary use Unix milliseconds.

## Cell Enrollment Grant

An enrollment grant is short-lived environment-root authority. Its canonical
body binds:

- contract and version.
- grant and enrollment nonce identity.
- environment key and current authority revision.
- active trust-root key, version, and public-key reference.
- cell key and trust profile: `trusted_control` or `standard`.
- cell control/containment public-key reference.
- distinct transport public-key reference and digest.
- fixed internal endpoint and `cadenza-cell-tls-v0` protocol.
- sorted capability grants with `use` or `admin` mode.
- positive slot capacity.
- issued and expiry times no more than five minutes apart.
- grant digest and root attestation.

The candidate cell signs the canonical grant digest with its transport private
key. Enrollment requires both the valid root grant and candidate possession
proof. The root private key and cell private keys never enter authority rows,
chamber frames, business context, evidence, or logs.

## Enrollment State

Enrollment status is `active`, `suspended`, or `revoked`:

- `active`: identity may authenticate, receive static assignment, and
  participate in route eligibility when runtime state also agrees.
- `suspended`: identity remains known but cannot authenticate a new peer
  session, activate work, receive assignment, publish ready residency, or route.
- `revoked`: terminal denial; the same enrollment identity cannot return.

Enrollment state is durable semantic authority. It is not cell-process
readiness.

## Cell Generation

Each cell-host process creates one immutable generation identity containing
environment, cell, and cell-generation keys. Runtime state is `starting`,
`ready`, `draining`, or `stopped`.

Restart creates a new generation. Runtime observations and later transport
envelopes bind the exact generation; numeric revisions from another generation
have no meaning.

## Runtime Slice Authority

Every approved source slice has one current authority record binding:

- environment, slice key, object key, and version key.
- lane and trust profile.
- exact source artifact and provenance.
- runtime and adapter requirement.
- sorted capability requirements.
- an execution-evidence policy selecting the required profile and declaring
  whether ordinary execution processing is enabled.
- sorted capability facades that bind each generated internal operation to one
  exact source-slice handler, capability, and `use` or `admin` mode.
- runtime-support digest.
- activation policy and status.
- artifact revocation state.

The bootstrap authority-access slice is the privileged first record in this
general contract. It retains its unique direct `authority_access/admin`
restriction. Ordinary source slices cannot acquire that capability.

Execution-evidence processing is authority-bound rather than injected as a
generic callback. V0 admits only the fixed `execution_evidence` claim, renew,
release, and acknowledge operations and the fixed
`execution_evidence_ledger/append_batch` operation. Generated task and intent
identities remain internal to the slice; external callers cannot supply an
arbitrary handler, provider method, SQL statement, or capability name.

## Placement Unit

A placement unit contains:

- unit key and immutable definition revision.
- mode: `singleton` or `replicated`.
- positive slot cost.
- one or more sorted runtime-slice memberships.
- membership role: `primary` or `support`.
- derived sorted capability requirements.
- derived `requires_trusted_cell`.

All V0 members are required. One replica materializes one chamber per member,
so a multi-slice unit scales and becomes ready as one higher-order identity
without weakening the chamber's one-slice image boundary.

Derived requirements must exactly equal the union of current member
requirements. `admin` dominates `use` for one capability. A privileged member
requires a trusted-control cell.

## Static Replica And Assignment

A placement replica binds one stable replica key to one unit definition
revision. It has no address.

An assignment binds:

- replica key and unit key.
- assigned cell key.
- positive assignment epoch.
- expected unit definition revision.
- state: `assigned` or `withdrawn`.
- authority revision and evidence.

Assignment is accepted only when:

- enrollment is active.
- trust requirements are satisfied.
- every required capability and mode is granted.
- the sum of assigned unit slot costs does not exceed cell capacity.
- the unit, replica, cell, and revision identities agree.

Static authority can validate a supplied assignment. Sprint 7 adds a separate
desired-state and reconciliation path that chooses from current eligible Cells
without weakening this assignment contract.

## Desired State And Deterministic Reconciliation

Placement desired state declares bounded replica demand and constraints. It
does not contain process commands, endpoints, credentials, or runtime bundles.
PostgreSQL assigns separate semantic input and control revisions and advances
them only when their whole-relevant meaning changes.

One canonical snapshot combines desired state, current Cells, definitions,
replicas, assignments, overrides, and observations. The pure planner returns
one bounded canonical plan with exact ordered actions and explicit unsatisfied
demand. PostgreSQL commits the plan before any action can affect authority.
Each action is immutable, idempotent, lease-fenced, and independently reloads
current authority. Outcomes and applications remain durable so a lost response
cannot cause blind re-execution.

The singleton reconciliation stem is an eight-task source graph in a contained
trusted-control Chamber. It reads only six literal control facades, uses the
same pure planner, commits the result, and reaches action affect only through a
separate digest-pinned authority-access Chamber. The control credential cannot
execute actions and the source cannot mount authority access.

The stem lease is five minutes and renews only inside its final two minutes.
PostgreSQL notifications are bounded wake hints; the trusted Cell coalesces
them with a five-second safety tick into one active signal and one pending bit.
Equal semantic authority creates no plan, action, process, residency, route, or
lease churn.

## Stem Recovery And Fencing

Every ready trusted-control Cell resolves stem custody through a separate
host-only PostgreSQL role on notification hints and the existing safety cadence.
Database time and one locked transaction determine owner-generation loss or
lease expiry. The resolver validates exact generation, trust, capabilities,
artifacts, desired policy, overrides, and capacity before selecting the
canonical eligible successor.

Takeover advances the lease epoch and, when custody moves, the singleton
assignment epoch. It retires unresolved predecessor actions as
`lease_replaced`, advances authority revisions, and appends immutable evidence
in the same transaction. Old lease, generation, assignment, residency, and
plan identities cannot affect later authority.

`takeover_committed` is not runtime readiness. Ordinary convergence must
materialize both exact stem members and publish current ready residencies before
the authority mount and its activation policy move together. Only a later exact
`owner` result may deliver the private stem signal. Recovery never starts
dormant Cells; no eligible ready trusted Cell is an explicit operator recovery
condition.

## Pre-Enrolled Cell Supply

Reconciliation may request process capacity only from profiles registered
before planning begins. A profile binds one already enrolled Cell identity to
one provider, one pinned launch-profile digest, and either `retained` or
`demand` release policy. PostgreSQL stores no executable, arguments,
environment, path, endpoint secret, credential, descriptor, or mount body.

The pure planner exposes only Cell key, release policy, provider availability,
current directive, observed provider state, observed Cell generation key, and
normalized failure reason. It can request `running`, `draining`, or `stopped`;
it cannot claim that a process transition happened. Placement still requires
the independent current Cell-signed `ready` generation.

One provider generation holds process-supply custody at a time. Immutable,
signed provider events bind its generation epoch and exact profile-catalog
digest. Immutable directives bind the committed plan and action to one exact
provider/profile revision. Signed supply observations report bounded process
realization without replacing Cell generation authority.

The provider projection also reports the current unexpired Cell generation for
each directive. A restarted provider waits for that generation to stop or
expire; V0 never adopts a surviving child and never starts an overlapping
generation. Reading this high-frequency projection is non-mutating and does
not create operation-evidence churn.

The Linux provider receives only fixed inherited descriptors and realizes one
closed local profile catalog. Each profile binds the exact Cell executable,
host configuration template, keys, certificates, endpoints, purpose-separated
database credentials, and journal root by canonical manifest digest. Profiles
are opened beneath pre-established directory custody with no symlink traversal
and are reverified before every launch. The child has an empty environment, a
fixed working directory, fixed descriptors, no shell, and parent-death
termination. The provider does not adopt children or keep another desired-state
database.

Supply-managed Cells receive a separate literal read-only lifecycle authority.
Drain requires the exact current directive and an empty local inventory; stop
requires the exact stopped directive and prior signed draining state. The Cell
alone publishes `starting`, `ready`, `draining`, and `stopped` generation
authority. The lifecycle reader cannot access tables or mutate supply, and the
provider cannot publish Cell state.

Drain and stop are rechecked transactionally. They fail while the Cell owns an
assignment, non-stopped residency, current stem lease, or operational authority
mount. Automatic release applies only to empty `demand` profiles. Provider
credentials cannot enroll Cells, change desired state, plan, assign replicas,
publish Cell generations, execute authority actions, or read private tables.

## Replica Residency

Residency is signed observed runtime state from the assigned cell. It binds:

- environment, unit, replica, assignment epoch, and cell generation.
- exact member slice, chamber, image digest, and image epoch identities.
- status: `materializing`, `ready`, `draining`, or `stopped`.
- observation time, expiry, digest, transport-key identity, and signature.

A replica is ready only when every required unit member has one unexpired ready
residency for the current assignment epoch and cell generation. Residency cannot
change assignment authority.

## Activation And Containment Lifetimes

An operational activation grant is short-lived admission authority. It is
capped at five minutes and must be live when the Cell activates the exact
Chamber image.

Successful admission creates a separate signed containment plan. Chamber
process custody is independently capped at fifteen minutes and is enforced by
both the Cell host and launcher. Grant expiry after admission does not
terminate an otherwise valid Chamber. Assignment, projection, residency,
route, and containment authority continue to govern its affect. No caller can
extend custody beyond the signed plan after activation.

## Operational Authority-Mount Promotion

Ready residency does not by itself authorize authority-gateway admission. The
activation issuer may promote the environment's singleton operational mount
only for the `slice.authority-access.v0` support member of
`unit.meta.reconciliation-stem.v0` and
`replica.meta.reconciliation-stem.0001`.

Promotion requires one exact current chain: active trusted Cell and generation,
current singleton assignment, signed operational activation grant, unexpired
signed-ready residency, current runtime-slice materialization, unrevoked
authority artifact, and the existing mount anchor. The transaction preserves
the mount key, authority slice, and artifact while replacing the Cell, Chamber,
image, epoch, and operational-state identity. Every actual transition produces
immutable canonical promotion evidence. Replays cannot create another
transition.

Only the UID-bound operational activation issuer can invoke this operation.
The Cell, Chamber, observation appender, reconciliation stem, and authority
gateway receive no direct mount-mutation privilege.

## Route Member

A route member is a derived projection, not authored state. It binds:

- route-member key derived from replica, owning slice, and responsibility so
  sibling slices declaring the same signal cannot collide.
- unit and immutable unit-definition revision.
- replica, assignment epoch, cell, and cell-generation identity.
- destination member slice, lane, chamber, image digest, and image epoch.
- task or signal responsibility.
- route epoch.

Eligibility requires active enrollment, current assignment, complete ready
residency, and responsibility declared by the current source slice. Cell
endpoints and keys remain in the cell projection; chamber projections receive
only opaque route-member keys.

Route groups and their members have one canonical sort order. Empty route
groups are absent rather than represented differently by PostgreSQL and Cell
projections.

## Authority Operations

V0 adds these fixed operations:

- `RuntimeSlice.Register`
- `RuntimeSlice.Revoke`
- `Cell.Enroll`
- `Cell.Suspend`
- `Cell.Revoke`
- `Placement.DefineUnit`
- `Placement.DefineReplica`
- `Placement.AssignReplica`
- `Placement.PublishResidency`
- `Reconciliation.ExecuteAction`

Operations are strict, idempotent, revision-bound, transactional, and
evidence-producing. There is no generic enrollment, placement, or SQL method.

Reconciliation control additionally exposes literal private operations for
work read, same-owner lease renewal, snapshot issue, plan commit, known action
failure, and changed-authority action closure. These are capability facades,
not public authority operations, and their role cannot execute a placement
action.

## Distribution-Control Meta Slice

The management API is an ordinary declarative Cadenza meta slice. Its tasks
delegate to exact authority-access task identities and receive neither a
database credential nor `authority_access/admin`. Multi-step static placement
is a visible task relationship from unit definition through replica definition
to assignment. Enrollment proof verification remains a cell-host security
operation before the meta slice receives the brokered proof reference.

The V0 fixture is the neutral source authority for this graph. Sprint 6B may
materialize and contain it, but may not replace it with a generic management
service or hidden provider call chain.

## Projection Reader

A separate least-privilege reader returns only the current cell's bounded
distribution projection:

- self enrollment and generation expectations.
- assigned units and replicas.
- current peer enrollment public data and fixed endpoints needed by eligible
  routes.
- derived local and remote route members.
- projection revision and digest.

It returns no private key, credential, provider, generic authority row, or
unrelated environment topology.

## Evidence

Enrollment, lifecycle, placement, assignment, residency, and projection changes
produce immutable evidence with request, idempotency, authority revision,
subject, runtime principal, outcome, and canonical result digest. Raw private
keys, credentials, signatures from unrelated domains, and business context are
not evidence facts.

Realtime business execution evidence is governed by the separate
`execution-evidence` contract. This distribution contract does not disguise
authority/runtime evidence as that stream or redefine its causal identities.

## Explicit Non-Goals

- cell creation or cloud-provider APIs.
- dynamic enrollment or construction of new launch profiles.
- endpoint mutation or transport-key rotation.
- public ingress, relays, or cross-environment trust.
- actors, persistence, memory, agents, UI, or CLI.
- concurrent reconciliation planning, action dispatch, or Chamber lifecycle.
