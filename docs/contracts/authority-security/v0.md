# Authority And Security Contracts V0

Date: 2026-07-12

## Purpose

This contract is the first language-neutral authority and security contract for Sprint 2.

It defines the semantic vocabulary for logical objects, immutable versions, tags, effective tags, policy decisions, and canonical authority flows before those concepts are expressed as TypeScript APIs, Python APIs, Elixir modules, C# types, database tables, or chamber/cell enforcement rules.

TypeScript remains the first proving implementation. This document and the fixtures beside it define the shared meaning.

## Authority Mode

The authority model for Sprint 2 is split:

- working implementation authority: `cadenza`
- semantic authority: language-neutral specs, fixtures, invariants, and decision records
- later durable authority: environment database records and projections

The core repos must not treat database storage, source materialization, cells, chambers, or runtime adapters as part of this contract.

## Naming And Identity Rules

- Logical objects use stable `object_key` values.
- Object keys are lowercase dot-separated or hyphenated semantic identifiers such as `task.orders.validate`.
- Object key segments begin with lowercase letters.
- Names and display labels are metadata, not identity.
- Versions are immutable and use stable `version_key` values.
- Tags use stable `tag_key` values.
- Tag categories use stable `category_key` values.
- Policy decisions use stable `decision_key` values when persisted or emitted as evidence.
- General authority keys are lowercase dot-separated segments with optional hyphens.
- Intent names in semantic payloads are lowercase hyphen-separated names.

## Logical Object Contract

A logical object is the authority identity for a Cadenza-relevant thing.

Minimum fields:

- `object_key`
- `object_type`
- `display_name`
- `lifecycle_state`
- `taggable`
- `policy_subject`
- `policy_resource`

Invariants:

- `object_key` is unique.
- primitive payload does not live directly in the logical object identity record.
- lifecycle state must be explicit.
- an object may be a policy subject, a policy resource, both, or neither.
- taggability must be explicit because tags carry authority consequences.

## Version Contract

A version is an immutable semantic payload for a logical object.

Minimum fields:

- `version_key`
- `object_key`
- `content_hash`
- `semantic_payload`
- `created_at`

Invariants:

- versions are immutable after creation.
- `content_hash` represents the canonical semantic payload.
- `(object_key, content_hash)` uniquely identifies semantic version content for one logical object.
- current selection is not stored on the version; it is represented by a version pointer.
- rollout/review state uses version marks, not ordinary tags.

## Version Pointer Contract

A version pointer selects a version for a purpose.

Minimum fields:

- `object_key`
- `pointer_kind`
- `version_key`

Initial pointer kinds:

- `current`

Invariants:

- a pointer must reference a version for the same object.
- changing a pointer is a distinct semantic action from creating a version.

## Version Mark Contract

A version mark annotates a version for rollout, review, or audit.

Minimum fields:

- `mark_key`
- `version_key`
- `mark`
- `primary`

Initial marks:

- `candidate`
- `stable`
- `deprecated`

Invariants:

- marks are separate from the main tag system.
- V0 permits at most one primary mark per version because no separate primary-mark family is modeled yet.

## Tag Contract

Tags are governed grouping and authority inputs, not decorative labels.

Minimum tag category fields:

- `category_key`
- `display_name`
- `cardinality`
- `managed_by`

Initial cardinality values:

- `single`
- `multi`

Minimum tag fields:

- `tag_key`
- `category_key`
- `display_name`

Invariants:

- direct tag assignment must preserve category cardinality.
- system-managed categories require explicit tag-management authority.
- tags can affect materialization, visibility, ownership, and policy decisions.

## Direct Tag Assignment Contract

A direct assignment is the authority record that a tag has been assigned to one logical object.

Minimum fields:

- `assignment_key`
- `object_key`
- `tag_key`
- `category_key`
- `reason`
- `assigned_by`

Invariants:

- copied `category_key` is intentional so the assignment can be interpreted historically.
- at most one active assignment may exist for the same object and tag.
- a `single` category permits at most one active assignment for an object in that category.
- direct assignment can be strict or bounded, depending on the flow.
- effective tags may be recomputed reactively after the direct assignment commits.

## Direct Tag Removal Contract

A direct removal is immutable evidence that one historical direct assignment no longer contributes to effective authority.

Minimum fields:

- `removal_key`
- `assignment_key`
- `object_key`
- `tag_key`
- `category_key`
- `reason`
- `removed_by`

Invariants:

- removal references an existing direct assignment for the same object, tag, and copied category.
- removal does not erase the original assignment evidence.
- removed assignments must not remain provenance sources for current effective tags.
- effective-tag recomputation may converge after the removal commits.

## Effective Tag Contract

Effective tags are normalized projections with provenance.

Minimum fields:

- `object_key`
- `tag_key`
- `sources`

Each source includes:

- `source_type`
- `source_key`

Invariants:

- policy should evaluate normalized effective tags first.
- cache fields may exist later, but caches must not become semantic authority.
- effective tag provenance must explain why a tag is effective.
- every provenance assignment must belong to the same object and tag as the effective-tag projection.

## Policy Decision Contract

Policy is allow-only and default-deny.

Minimum fields:

- `decision_key`
- `family`
- `action`
- `subject_object_key`
- `resource_object_key` or `target_tag_key`
- `target_object_key` for tag-management decisions
- `outcome`
- `matched_rule_keys`
- `reasons`

Initial decision families:

- `resource_access`
- `tag_management`

Initial resource actions:

- `read`
- `write`
- `execute`
- `materialize`

Initial tag-management actions:

- `assign_tag`
- `remove_tag`
- `create_tag`

Outcomes:

- `allow`
- `deny`

Invariants:

- absence of a matching allow rule means deny.
- decisions should carry enough explanation to be evidence later.
- allow decisions must include at least one matched rule.
- every matched rule must have the same policy family and action as the decision.
- deny decisions may have no matched rules but must include reasons.
- tag-management decisions must identify both the target object and target tag.
- policy evaluates effective tags, lifecycle state, object type, access mode, and other authority context as the contract grows.

## Canonical Flow Families

Initial canonical flows:

- `Version.CreateInitialObject`
- `Version.CreateNextVersion`
- `Version.SetCurrent`
- `Tag.Assign`
- `Tag.Remove`
- `Tag.RecomputeEffectiveForObject`
- `Policy.EvaluateResourceAction`
- `Policy.EvaluateTagAction`

V0 conformance cases must declare every canonical flow exactly once.

Strictness:

- `Version.CreateInitialObject`: strict
- `Version.CreateNextVersion`: bounded
- `Version.SetCurrent`: strict or bounded depending on rollout policy
- `Tag.Assign`: bounded
- `Tag.Remove`: bounded
- `Tag.RecomputeEffectiveForObject`: loose
- `Policy.EvaluateResourceAction`: strict for authority gates, otherwise bounded as evidence
- `Policy.EvaluateTagAction`: strict for tag mutation gates

Minimum flow declaration fields:

- `flow_key`
- `shape`
- `strictness`
- `semantic_boundary`
- `invariant_fields`
- `deferred_consequences`

Initial shapes:

- `orchestrated`
- `reactive`

Invariants:

- strict flows must state the invariant fields that must succeed or fail together.
- bounded flows must state which consequence is immediate and which consequences may converge later.
- loose flows must state why eventual convergence is sufficient.
- no flow declaration may imply direct task-to-task calls from inside task logic; request/response work uses intents and detached work uses signals.

## Flow Request And Result Envelopes

Canonical flow envelopes are neutral evidence shapes for requests and results. They describe what a controlled runtime or later environment layer is allowed to attempt and what committed semantic result it may report.

They are not executable handlers, database commands, source materializers, or chamber/cell implementations.

Common envelope fields:

- `flow_key`
- `request`
- `result`

Common request fields:

- `request_key`
- `caller_object_key`

Creation identity rule:

- requests name every new authority identity they may establish.
- implementations validate proposed identities but do not replace them with
  provider-generated identities.
- unused proposals create no authority when a flow rejects, reuses existing
  authority, or resolves to no change.
- derived content digests must be recomputed and verified by the trusted broker
  or authority provider rather than accepted as unverified caller authority.

Common result fields:

- `outcome`
- `deferred_consequences`

### Version.CreateInitialObject Envelope

Minimum request fields:

- `request_key`
- `object_key`
- `object_type`
- `display_name`
- `initial_lifecycle_state`
- `taggable`
- `policy_subject`
- `policy_resource`
- `version_key`
- `mark_key`
- `initial_version_payload`
- `initial_version_mark`
- `caller_object_key`

Minimum result fields:

- `outcome`
- `object_key`
- `version_key`
- `current_version_key`
- `created_keys`
- `deferred_consequences`

Initial outcomes:

- `created`
- `rejected`

Invariants:

- successful results must report the same `object_key` requested.
- successful results must preserve the requested object authority properties.
- successful results must use the proposed version and mark identities.
- successful results must identify the immutable version created and the current pointer target.
- `created_keys` must contain the semantic authority records created by the flow.
- rejected results must not include `created_keys`.
- this envelope does not materialize callable source strings.

### Tag.Assign Envelope

Minimum request fields:

- `request_key`
- `object_key`
- `tag_key`
- `assignment_key`
- `authorization_decision_key`
- `reason`
- `caller_object_key`

Minimum result fields:

- `outcome`
- `assignment_key`
- `object_key`
- `tag_key`
- `category_key`
- `deferred_consequences`

Initial outcomes:

- `assigned`
- `rejected`

Invariants:

- successful results must report the same object and tag requested.
- successful results must use the proposed assignment identity.
- successful results require a matching `allow` tag-policy decision bound to
  the authority revision being mutated.
- rejected results preserve explicit deny, stale, missing, or mismatched
  authorization interpretation without creating assignment authority.
- successful results must preserve the tag category copied onto the direct assignment.
- rejected results may name the requested object, tag, copied category, and rejected assignment key without requiring a committed direct assignment record.
- effective tag recomputation is a deferred consequence unless the implementing environment explicitly tightens the flow.
- this envelope does not enforce storage, placement, or runtime execution policy by itself.

### Version.CreateNextVersion Envelope

Minimum request fields:

- `request_key`
- `object_key`
- `base_version_key`
- `proposed_version_key`
- `proposed_mark_key`
- `semantic_payload`
- `candidate_mark`
- `caller_object_key`

Minimum result fields:

- `outcome`
- `object_key`
- `base_version_key`
- `observed_current_version_key`
- `resolved_version_key`
- `created_keys`
- `deferred_consequences`

Outcomes:

- `created`
- `reused_existing`
- `no_change_current`
- `rejected_stale_base`

Invariants:

- the base version, observed current version, and any resolved version must belong to the requested object.
- non-rejected results require the submitted base version to equal the observed current version.
- `created` resolves to a new immutable version, reports its created authority keys, and proves the requested candidate mark exists.
- `created` uses the proposed version and mark identities.
- `reused_existing` resolves to an existing non-current version and creates no authority records.
- `no_change_current` resolves to the observed current version and creates no authority records.
- `rejected_stale_base` reports the actual observed current version, has no resolved version, and creates no authority records.
- this flow never moves the current pointer.

### Version.SetCurrent Envelope

Minimum request fields:

- `request_key`
- `object_key`
- `expected_current_version_key`
- `target_version_key`
- `reason`
- `caller_object_key`

Minimum result fields:

- `outcome`
- `object_key`
- `previous_current_version_key`
- `current_version_key`
- `pointer_changed`
- `deferred_consequences`

Outcomes:

- `updated`
- `no_change`
- `rejected_stale_pointer`

Invariants:

- expected, target, previous, and resulting current versions must belong to the requested object.
- `updated` requires the expected version to match the previous pointer, the target to become current, and `pointer_changed` to be true.
- `no_change` requires expected, target, previous, and current versions to agree and `pointer_changed` to be false.
- `rejected_stale_pointer` reports the observed current pointer without changing it.
- V0 treats pointer mutation as a strict semantic boundary; audit export and projections may remain deferred.

### Tag.Remove Envelope

Minimum request fields:

- `request_key`
- `object_key`
- `tag_key`
- `assignment_key`
- `removal_key`
- `authorization_decision_key`
- `reason`
- `caller_object_key`

Minimum result fields:

- `outcome`
- `removal_key`
- `assignment_key`
- `object_key`
- `tag_key`
- `category_key`
- `deferred_consequences`

Outcomes:

- `removed`
- `rejected`

Invariants:

- the request must reference an existing direct assignment for the requested object and tag.
- successful results must bind to immutable direct-removal evidence for the same assignment, object, tag, category, reason, and caller.
- successful results must use the proposed removal identity.
- successful results require a matching `allow` tag-policy decision bound to
  the authority revision being mutated.
- rejected results do not require committed removal evidence.
- effective-tag and provenance recomputation remain deferred consequences.

### Tag.RecomputeEffectiveForObject Envelope

Minimum request fields:

- `request_key`
- `object_key`
- `caller_object_key`

Minimum result fields:

- `outcome`
- `object_key`
- `effective_tag_keys`
- `provenance_source_keys`
- `deferred_consequences`

Outcomes:

- `recomputed`
- `no_change`
- `rejected`

Invariants:

- successful and no-change results must agree with the normalized effective-tag and provenance records for the object.
- rejected results must not claim effective tags or provenance sources.
- this loose flow projects authority; it does not become authority over direct assignments or removals.

### Policy.EvaluateResourceAction Envelope

Minimum request fields:

- `request_key`
- `subject_object_key`
- `resource_object_key`
- `action`
- `decision_key`
- `caller_object_key`

Minimum result fields:

- `outcome`
- `decision_key`
- `matched_rule_keys`
- `reasons`
- `deferred_consequences`

Outcomes:

- `allow`
- `deny`

Invariants:

- successful normalization must prove the referenced subject and resource objects exist.
- the result must reference a policy decision for the same subject, resource, and action.
- the result must use the proposed decision identity.
- the result outcome, matched rules, and reasons must agree with the referenced policy decision.
- absence of a matching allow decision remains a denial concern for the runtime/environment layer; this envelope only records the semantic decision evidence it is given.

### Policy.EvaluateTagAction Envelope

Minimum request fields:

- `request_key`
- `subject_object_key`
- `target_object_key`
- `target_tag_key`
- `action`
- `decision_key`
- `caller_object_key`

Minimum result fields:

- `outcome`
- `decision_key`
- `matched_rule_keys`
- `reasons`
- `deferred_consequences`

Outcomes:

- `allow`
- `deny`

Invariants:

- subject, target object, target tag, and caller must exist.
- the result must reference a tag-management policy decision for the same subject, target object, target tag, and action.
- the result must use the proposed decision identity.
- result outcome, matched rules, and reasons must agree with the referenced policy decision.
- tag-policy evidence does not itself mutate tags.

## Translation Impact

All official language cores should be able to represent these contracts as data without adding runtime substrate responsibilities.

Conformance is defined in [conformance-v0.md](https://github.com/cadenzaio/cadenza-workspace/blob/main/docs/contracts/authority-security/conformance-v0.md).

The consolidated choreography catalog is [canonical-flows-v0.md](https://github.com/cadenzaio/cadenza-workspace/blob/main/docs/contracts/authority-security/canonical-flows-v0.md).

Translation notes:

- TypeScript should implement the first validator and fixture proof.
- Python should express the same contract with dataclasses or typed dictionaries later.
- Elixir should express the contract with maps/structs and atoms only for local field tags, not dynamic authored keys.
- C# should express the contract with records or sealed classes later.

No language core should compile or materialize source strings as part of this contract.
