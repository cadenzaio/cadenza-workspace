# Environment Bootstrap Contract V0

Date: 2026-07-12

## Purpose

This contract defines the smallest static transition from an empty authority store to a trustworthy Cadenza environment handoff.

It governs genesis evidence. It does not execute Cadenza graphs, materialize callables, activate plugins, reconcile placement, or claim that an environment is operational.

## Intended Whole

Bootstrap creates the minimum trustworthy authority state from which Cadenza can later extend itself through governed primitives.

The static root must become less powerful as bootstrap progresses. It exists to establish legitimate authority, not to remain a parallel management system.

## Contract Authority

Semantic authority consists of:

- this contract.
- `conformance-v0.md`.
- `fixtures/v0-bootstrap-foundation.json`.
- the approved environment-bootstrap decision.
- the existing authority/security V0 contract for seeded graph records.

TypeScript and PostgreSQL are first implementations, not semantic authority by themselves.

## Bootstrap States

V0 states are ordered:

1. `uninitialized`
2. `schema_ready`
3. `trust_established`
4. `foundation_seeded`
5. `authority_slice_ready`
6. `handoff_ready`

Invariants:

- transitions move only to the immediately following state.
- committed states never move backward.
- every committed transition has exactly one step-evidence record for its bootstrap run.
- `operational` is not a V0 bootstrap state.
- repair and trust-root rotation do not reuse this state machine.

## Environment Identity

Minimum fields:

- `environment_key`
- `lifecycle_state`
- `created_at`

V0 lifecycle state is `active`.

`environment_key` is a lowercase dot-separated authored identity. It is immutable after `trust_established`.

## Trust Root

Minimum fields:

- `environment_key`
- `root_key_id`
- `root_public_key_ref`
- `trust_root_version`
- `status`
- `created_at`

V0 status is `active` and V0 starts at version `1`.

Invariants:

- the record contains public identity/reference data only.
- fields named or shaped as private keys, secrets, credentials, tokens, or raw key material are forbidden.
- one active trust-root version exists for an environment after `trust_established`.
- trust-root rotation is a separate future authority flow.

## First Cell Enrollment

Minimum fields:

- `cell_key`
- `environment_key`
- `trust_profile`
- `enrolled_by`
- `cell_public_key_ref`
- `transport_public_key_ref`
- `transport_key_digest`
- `internal_endpoint`
- `transport_protocol`
- sorted `capability_grants`
- positive `slot_capacity`
- `status`
- `enrolled_at`

V0 requires:

- `trust_profile`: `trusted_control`
- `enrolled_by`: `static_bootstrap`
- `status`: `active`
- the control/containment and transport key references are distinct.
- `transport_protocol`: `cadenza-cell-tls-v0`.
- the trusted-control cell receives the governed `authority_access/admin`,
  `cell_control/admin`, `execution_evidence/use`,
  `execution_evidence_ledger/use`, `network_client/use`, and
  `network_server/use` grants.

The enrollment establishes identity and future eligibility. It does not prove
that a cell process is running. The endpoint is distribution authority, not a
readiness assertion.

## Bootstrap Run

Minimum fields:

- `bootstrap_run_key`
- `environment_key`
- `bootstrap_version`
- `input_digest`
- `requested_at`
- `state`

Invariants:

- one environment may have only one genesis input digest.
- rerunning the exact digest resumes or returns the existing result.
- a different digest after any committed genesis state is `trust_conflict`.
- generated timestamps and evidence keys do not participate in the normalized input digest.

The normalized bootstrap input includes:

- bootstrap version.
- bootstrap principal key and external proof reference.
- environment key.
- root key id and public-key reference.
- first-cell identity, trust profile, and public-key reference.
- ordered seed-pack identities, versions, and content digests.

## Bootstrap Step Evidence

Minimum fields:

- `evidence_key`
- `bootstrap_run_key`
- `environment_key`
- `sequence`
- `from_state`
- `to_state`
- `outcome`
- `recorded_at`
- `details_digest`
- `bootstrap_principal_key`
- `previous_evidence_digest`
- `evidence_digest`

V0 committed-step outcome is `committed`.

Invariants:

- sequence is positive and strictly monotonic per environment.
- state pair is one legal adjacent transition.
- evidence is append-only.
- the first step uses `null` as `previous_evidence_digest`; every later step references the preceding step's `evidence_digest`.
- `evidence_digest` covers canonical JSON of every evidence field except `evidence_digest`.
- evidence and state transition commit atomically.
- failure evidence may be recorded separately but cannot claim a state transition committed.

## Seed Pack

A seed pack is an immutable content-addressed envelope.

Minimum fields:

- `pack_key`
- `pack_version`
- `content_digest`
- `records`

Each record includes:

- `record_type`
- `record_key`
- `payload`

V0 pack keys:

- `seed.foundation.v0`
- `seed.authority-kernel.v0`

Invariants:

- `(pack_key, pack_version)` is unique.
- `record_key` is unique within a pack.
- `content_digest` covers canonical JSON of `{pack_key, pack_version, records}`.
- records are applied in listed order.
- an applied pack cannot be changed in place; changed content requires a new version and digest.
- every applied record traces to its seed application and bootstrap run.
- source text, inline executable code, private key material, credentials, and secret values are forbidden anywhere in seed payloads.
- handler references use pinned artifact identity, digest, and handler key.

## Bootstrap Handoff

Minimum fields:

- `handoff_key`
- `environment_key`
- `bootstrap_run_key`
- `bootstrap_state`
- `authority_revision`
- `trust_root`
- `first_cell`
- `authority_slice`
- `seed_packs`
- `runtime_requirement`
- `activation_policy_ref`
- `evidence_ref`
- `created_at`
- `handoff_digest`
- `attestation`

V0 requires `bootstrap_state` to be `handoff_ready`.

The handoff includes the final `evidence_chain_digest` alongside `evidence_ref`.

The handoff digest covers canonical JSON of all handoff fields except `handoff_digest` and `attestation`.

The attestation includes:

- `signing_key_id`
- `algorithm`
- `signature`

The signature covers the UTF-8 `handoff_digest` string. `signing_key_id` must equal the active trust-root key id. V0 permits `ed25519` as the attestation algorithm. Signature verification uses external key custody resolved through the trust-root public-key reference.

Conformance fixtures may provide public-key resolution data outside the authority case. Private keys must never appear in fixtures or authority records.

The future chamber/cell boundary must reject a handoff when:

- state is not `handoff_ready`.
- environment, trust-root, cell, slice, seed, artifact, runtime, capability, authority revision, or evidence identity is absent.
- any digest is malformed or mismatched.
- the evidence chain is discontinuous or its final digest differs from `evidence_chain_digest`.
- handoff attestation is missing, uses another key, or fails verification.
- trust-root or cell enrollment is not active.
- required capability or trust profile cannot be satisfied.
- referenced authority state has advanced incompatibly or been revoked.

The handoff contains no database credentials, private keys, provider tokens, source strings, or secret values.

## Canonical JSON V0

Canonical digest input uses a deliberately restricted JSON domain:

- UTF-8 encoding without byte-order mark.
- values may be objects, arrays, strings, booleans, null, or integers within the interoperable safe range `-(2^53-1)` through `2^53-1`.
- floating-point and exponential-number forms are forbidden in digest-bearing contracts.
- object keys must be ASCII and are sorted by ascending byte value.
- arrays preserve authored order.
- strings contain Unicode scalar values only; unpaired UTF-16 surrogates are invalid.
- strings use JSON escaping with no insignificant whitespace.
- duplicate object keys are invalid before canonicalization.

Digest form is `sha256:` followed by 64 lowercase hexadecimal characters.

This bounded rule exists so every future implementation can reproduce bytes exactly without inheriting language-local map ordering or number behavior.

Implementations may preserve native diagnostic details but must expose one of these stable classes at the contract boundary.

## Persistence Boundary

The first adapter uses PostgreSQL, but this contract does not expose table names, SQL, transaction APIs, or provider credentials.

Persistence must enforce:

- bootstrap state and evidence ordering.
- trust-root and first-cell identity uniqueness.
- input-digest idempotency and conflict rejection.
- immutable seed identity and application uniqueness.
- the authority/security V0 relational invariants for seeded graph records.
- one initial direct holder of `authority_access`.

## Non-Goals

- callable materialization or execution.
- chamber or cell process implementation.
- general authentication/session providers.
- plugin activation.
- secrets storage or provider implementation.
- stem-cell reconciliation, placement, distribution, or scaling.
- agent API, UI, CLI, memory, or business-graph seeds.
- legacy database migration.
- trust-root rotation or disaster-recovery ceremony.

## Translation Impact

No core-language translation is required. Static bootstrap is substrate code.

Every future store adapter, chamber runtime, or external bootstrap implementation that consumes this boundary must conform to the neutral JSON identities, canonical digest rules, state transitions, seed semantics, and handoff validation requirements.
