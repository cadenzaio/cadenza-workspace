# Actor Assignment And Residency Contract V1

## Purpose

This contract defines single-owner assignment, assignment-epoch fencing, local
Chamber residency, generated actor owner endpoints, and Sprint 8C durable
state operations.

An assignment is for one `(actor_object_key, actor_state_key_digest)` pair.
The digest is derived from the UTF-8 runtime state key as `sha256:` plus 64
lowercase hexadecimal characters. The raw state key is business-sensitive and
is absent from assignment rows, routes, endpoint identities, logs, and
evidence.

## Authority Split

- PostgreSQL owns actor placement authority, the current assignment pointer,
  and immutable assignment epochs.
- the Cell owns exact owner resolution, raw-key custody during execution,
  generated endpoint routing, and mapping an assignment to a local Chamber.
- the Chamber owns bounded local residency and rejects an invocation unless
  its assignment epoch, Cell generation, Chamber identity, image, and endpoint
  provenance are current.
- the primitive core remains persistence and topology agnostic.

Replica placement does not imply actor-key ownership. Actor assignment points
to one currently eligible placed replica and its ready actor-owning Chamber.

## Actor Placement Authority

Before first touch, authority binds one actor object to one current placement
unit revision and one actor-owning source slice. The binding contains no state
key, replica, Cell, Chamber, endpoint, or database provider identity.

The binding is established by an exact administrative operation. A Cell
resolving a key cannot nominate or alter this placement.

## Owner Resolution

The request contains exactly:

- `environment_key`
- `actor_object_key`
- `actor_state_key_digest`
- requesting `cell_key` and `cell_generation_key`
- `resolved_at_unix_ms`

PostgreSQL serializes resolution for the actor/key pair. It returns the current
assignment when that exact owner remains a current ready route member of the
bound placement unit. Additional capacity does not move a healthy assignment.

When there is no current valid owner, PostgreSQL chooses from the canonical
eligible ready replica set. The requester cannot nominate a candidate. The
choice is the highest stable SHA-256 rendezvous rank over environment, actor,
state-key digest, placement revision, and replica identity; replica identity
is the deterministic tie-breaker.

The selected assignment epoch is immutable. A separate current pointer may
advance to a later epoch after owner invalidation. An earlier epoch never
regains authority.

The resolution returns assignment identity and epoch, placement revision,
owner replica and replica-assignment epoch, Cell/generation, slice, Chamber,
image, and route identity, a `continued` or `created` outcome, and bounded
authority expiry. No eligible owner is an explicit unavailable outcome.
Database unavailability cannot become local ownership.

## Local Residency Grant

The owner Cell derives one exact grant from the resolution. It binds the actor
object and state-key digest to the assignment, replica, Cell generation,
Chamber, image, and actor-owning source slice. The grant expires no later than
the source assignment resolution.

The Chamber accepts a grant only when every identity matches its current image
and generation. For one actor/key pair it accepts the same grant idempotently
or a strictly newer epoch. It rejects older epochs, same-epoch identity drift,
foreign Cells, foreign Chambers, foreign images, unknown actor definitions,
and expired authority before handler execution.

Distributed residency uses strict write-through persistence. The conservative
Sprint 8 runtime cleanly unloads each distributed state instance after its
execution attempt and hydrates again on the next invocation; it does not retain
an unbounded adapter state cache behind the bounded Chamber registry. A clean
unload does not end global assignment, and a same-owner reload reuses the
current epoch while hydrating the latest committed state. After assignment
succession, the new owner hydrates that same durable state under a strictly
later epoch.

## Generated Owner Endpoint

For each task with `cadenza.actor.binding.v1`, the Chamber image contains one
hidden generated endpoint descriptor. It binds the authored task and version,
actor object, access mode, generated endpoint task identity, source slice,
Chamber, and image epoch provenance. The descriptor is itself inside the
canonical hashed image, so it does not contain the image digest and create a
self-referential hash. The residency grant and invocation carry the final
image digest checked by the Chamber.

The generated task uses the authored handler and actor binding but has no
authored relationships, observed signals, emitted-signal declarations,
handled intents, or independent placement meaning. It is not a public intent.

A non-owner execution delegates only to this endpoint. The endpoint returns
the local handler result. The caller's original task wrapper returns that
result to its original graph, so downstream relationships continue exactly
once in the caller.

## Durable State Operations

Distributed actors use three purpose-separated persistence operations:

- hydration reads current committed state under an exact current residency
  grant and returns either `absent` at version `0` or one committed object with
  its canonical digest and positive version.
- commit binds one stable `actor_mutation_key` to an expected state version,
  next state, handler result, and their canonical digests. Exact reuse returns
  `replayed`; reuse with different content is rejected.
- outcome resolution answers only whether that exact mutation committed and,
  when it did, returns its bounded result commitment. It exists for unknown
  transport outcomes and cannot read actor state generally.

Raw actor state keys never cross these contracts. Durable state and handler
results are bounded private execution data and are excluded from ordinary
routing, evidence, and diagnostics. `absent` is not equivalent to invalid,
oversized, or schema-incompatible state; those conditions fail explicitly.

Only the current owner Cell generation and Chamber image named by the
residency grant may hydrate or commit. State versions are monotonic across
assignment epochs. Assignment succession changes who may commit, not the
logical state identity or version history.

Each private persistence request also carries the trace, graph-execution, and
task-execution ancestry that caused it. PostgreSQL validates but does not use
that ancestry as state authority. The Cell uses it to produce bounded evidence
without recording raw state, state keys, input, or results.

## Drain And Relinquishment

A Chamber drain first rejects new actor work and proves that its bounded local
residency lanes have no in-flight operations. The owner Cell then invokes one
exact PostgreSQL relinquishment operation for that Cell generation, Chamber,
image digest, and image epoch. PostgreSQL removes only matching current
assignment pointers. The scoped Cell role may invoke this operation only after
the local Chamber residency and commit barriers complete; PostgreSQL does not
infer that local barrier from lagging residency observations. Immutable
assignment epochs and durable state remain.

The next touch may create a strictly later epoch on an eligible successor and
hydrate the last committed state. Relinquishment is idempotent and returns a
bounded aggregate count; it never returns actor state keys or state.

## Actor Lifecycle Evidence

Cell-observed lifecycle evidence covers assignment, routing, hydration,
commit, unknown-outcome resolution, and aggregate drain relinquishment. It
includes definition identity, state-key digest, epoch, versions, mutation
identity, generation and image identity, endpoint provenance digest, result
and state commitments, and execution ancestry where an execution caused the
event.

Raw actor state keys, state, business input/output, credentials, connection
details, and generated endpoint payloads are excluded. Lifecycle-only drain
evidence has no execution ancestry or individual actor identity and reports
only the bounded affected-assignment count.

## Fencing Order

Before local actor affect, the runtime checks request and endpoint shape,
current Chamber lifecycle/image, Cell and Chamber generation, exact assignment
epoch and owner, residency grant, then strict per-key execution admission.

Changing actor state version cannot satisfy an assignment fence. Changing a
replica assignment epoch cannot satisfy an actor assignment fence.

## Bounds

- owner resolution accepts one actor/key pair and returns one owner.
- candidate owners are bounded to 4,096 and canonical and unique.
- one Chamber holds at most 65,536 active residency grants; idle entries are
  cleanly evictable and capacity admission reclaims an idle entry before
  rejecting pressure.
- one actor/key queue holds at most 1,024 waiting invocations.
- actor state and results have a maximum canonical depth of 64 in addition to
  their 1 MiB byte bounds.
- endpoint descriptors are bounded by the runtime image task bound.
- contract values use canonical JSON and safe positive integers.

## Required Proofs

- two Cells concurrently resolving an unassigned key receive the same epoch.
- repeated resolution retains a healthy owner after new capacity appears.
- invalid owner authority creates one strictly later epoch.
- stale epoch, generation, image, and endpoint provenance fail before affect.
- same-key execution is strict while distinct keys may progress independently.
- remote execution runs the handler once and caller relationships once.
- raw state keys are absent from assignment, route, endpoint, and evidence
  records.
