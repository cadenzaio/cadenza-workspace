# Distributed Foundation Stabilization And Publication

Date: 2026-07-21

## Context

Sprint 8 completed the first coherent distributed Cadenza foundation across
four core languages, environment authority, Cell containment and distribution,
Chamber materialization and execution, evidence custody, scale, supply, stem
succession, and durable actor lifecycle.

Publishing immediately would export unresolved repository boundaries,
historical artifacts, uneven public documentation, operational assumptions,
and an architecture that remains difficult to interpret visually. Focused
implementation tests also do not by themselves prove that an application
author can create realistic business behavior without absorbing distribution
and runtime complexity.

## Decision

Cadenza will complete a feature-free Sprint 9 before public publication. The
sprint is divided into:

1. truth baseline and publication scope.
2. recursive code, contract, and coherence review.
3. security, reproducibility, realistic system proof, and operational
   hardening.
4. a visual architecture atlas and public documentation.
5. release engineering and public packaging.
6. definitive release-candidate proof and controlled publication.

Diagrams are versioned interpretive contracts and review instruments. Draft
visuals begin during inventory, help reveal architectural fragmentation, and
are finalized only against reviewed release-candidate code and contracts.

Sprint 9C will build a release-owned order-to-fulfillment reference system with
multiple realistic business flows. It must use every current authored
primitive for a defensible business reason and exercise representative
authority, distribution, scale, evidence, failure, actor recovery, and cleanup
behavior. A clean-context agent authoring trial will test whether public
Cadenza surfaces actually keep attention on business logic and workflow shape.

The exact public repository set, documentation home, license, git-history
strategy, release maturity, contribution model, support posture, and package
registry scope remain explicit Sprint 9A judgment gates. No external
publication occurs without a separate approval after release-candidate review.

## Consequences

- Feature development remains frozen until publication closure.
- Cleanup is evaluated against the intended whole, not only local lint or test
  success.
- Every retained production artifact and public claim needs a defensible
  purpose and evidence.
- The reference system becomes both an outside-in release proof and a teaching
  source for public diagrams and documentation.
- Agent difficulty is treated as framework evidence; private explanation or
  better prompting cannot conceal an incoherent API or missing document.
- Public publication remains reversible until the final external push gate;
  preparation does not imply package-registry release.

## Alternatives

### Publish The Sprint 8 State Immediately

Rejected. Focused correctness does not prove public reproducibility,
interpretability, security posture, or usability.

### Treat Diagrams As Post-Release Documentation

Rejected. Drawing identities, boundaries, state, and consequence is part of
the architecture review and can reveal defects before release.

### Use Only Internal Integration Tests

Rejected. Internal tests can validate components while missing accidental
complexity experienced by an application author.

### Reuse A Legacy Demo

Rejected. Legacy repositories do not own the current architecture and would
reintroduce obsolete assumptions and glue.
